처음에는 제목을 "C++ 개발자가 Java에 적응하기까지"라고 지었다가 지금의 제목으로 고쳤다. 아직 Java에 적응을 마치지 못했기 때문에 "적응하기까지"라는 표현은 틀렸다.
팀과 업무와 사용 언어가 바뀐지 1년이 넘었건만 아직 적응 중이다. 어쩌면 영영 적응을 마칠 일이 없을지도 모른다.

내 지난 경험을 조금 풀어보겠다. Java를 처음 접했을 때가 대학교 2학년이었다. Java 2가 나온지 오래되지 않은 시점이었다.
이후 한두 과목에서 Java로 프로젝트를 했고, 졸업 후 한동안 개발과 동떨어진 일을 하다가 거의 5년 만에 짧은 인턴 기간 중 다시 접할 기회가 생겼다. 이때는 Java 5였다고 기억한다.

C++에 관한 기억은 조금 더 단순한데, 자료 구조 수업에서 사용한 이후로 죽 손댈 일이 없다가, 역시 인턴 기간에 잠시 기억을 되살릴 시간을 가졌다.
얼마 후 지금의 회사에서 검색 엔진 개발팀에 들어가면서 10년 넘게 C++을 곁에 두고 살았다.
그동안 C++ 역시 여러 차례 새 표준이 나오면서 꾸준히 변했다.

1년 반쯤 전, 새로운 팀에 들어가고 업무가 바뀌었다. 동시에 주 언어도 C++에서 Java로 바뀌었는데, 이미 현업에서는 Java 8 또는 그 이후 버전이 널리 퍼졌다.
과거에 알던 Java와는 많은 부분이 달라졌고, 익숙해지는데 시간이 좀 걸렸다.
다만 지금 얘기하려는 내용은 Java 2나 5에서 8으로 넘어가면서 겪은 생소함이 아니라 (생소함을 느끼기엔 5 경험조차 너무 오래 전 일이다) C++에서 익숙한 개념이 Java에는 달라서 겪은 실수다.

### 복사 생성자가 없다.

C++에는 클래스에 복사 생성자가 있어서 한 객체의 값을 다른 객체로 복사할 수 있다. 반면 Java는 복사 생성자가 없고, 변수 대입은 C++의 포인터 또는 참조자와 같이 동작한다.

아래와 같은 C++ 코드에서
```cpp
Foo a;
Foo b = a;
```

b와 a는 서로 다른 객체이므로, a의 값을 변경해도 b에는 영향이 없다. 반면 아래와 같은 Java 코드에서
```java
Foo a = new Foo();
Foo b = a;
```

a와 b는 같은 객체를 가리키며, 따라서 a의 값이 바뀌면 b의 값도 바뀐다.

참 단순한 차이인데, 복사 생성자에 익숙하다보니 a와 b가 다르다고 간주하는 실수를 겪곤 했다.

### primitive 대응 객체 형식은 nullable

Java에서는 어떤 변수가 new 연산자로 객체를 생성하지 않으면 null이다.
이건 primitive 형식에 대응하는 Boolean, Integer, Double 등도 마찬가지인데, 이들을 자꾸 primitive 형식인 bool, int, double로 혼동하곤 한다.

이들 객체를 읽을 때는 null 검사를 하거나 null 인자에 안전한 메소드를 사용해야 하는데, 특히 비교 연산자 사용시 이를 빠뜨릴 때가 있다.

```Java
Integer i;    // null!
...
if (i == 0) {    // NullPointException!
}
```
